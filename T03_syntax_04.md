## 주요 내용과 키워드

### 1. LL(1) 문법 (LL(1) Grammars)

LL(1)은 입력 스트림을 왼쪽에서 오른쪽으로 읽고(Left-to-right), 최좌측 유도(Leftmost derivation)를 수행하며, 1개의 토큰을 미리 본다(1 token look-ahead)는 의미입니다.

#### **LL(1)이 되기 위한 조건**

- **좌재귀(Left-recursion) 부재:** 자기 자신을 참조하는 재귀가 생성규칙의 왼쪽에 오면 안 됩니다.
- **공통 접두사(Common prefixes) 부재:** 동일한 비단말 기구에서 시작하는 여러 규칙의 시작 토큰이 겹치면 안 됩니다.
- **모호성(Ambiguity) 부재:** 하나의 문장에 대해 두 개 이상의 유도 트리가 존재하면 안 됩니다.

### 2. 예측 파싱 및 문법 변환

단일 토큰 look-ahead를 통해 다음에 확장할 생성 규칙을 정확히 선택하는 기법입니다.

- **좌재귀 제거(Left-recursion elimination):** 무한 루프를 방지하기 위해 $N ::= X | NY$ 형태를 $N ::= XY^*$ 형태로 변환합니다.
- **좌인수분해(Left-factorization):** 공통된 접두사를 가진 규칙들을 묶어 선택의 모호함을 해결합니다.
  - 예: $X Y | X Z \rightarrow X(Y | Z)$

### 3. FIRST와 FOLLOW 집합

LL(1) 문법 여부를 판단하고 파서를 구축하기 위한 핵심 도구입니다.

#### **FIRST($\alpha$)**

- 문자열 $\alpha$로부터 유도되어 나올 수 있는 첫 번째 단말 기호(Terminal)들의 집합입니다.
- 만약 $\alpha$가 빈 문자열($\epsilon$)을 생성할 수 있다면 $\epsilon$도 포함됩니다.

#### **FOLLOW(A)**

- 비단말 기호 $A$ 바로 뒤에 나타날 수 있는 단말 기호들의 집합입니다.
- 입력의 끝을 나타내는 `$`(EOF) 기호가 포함될 수 있습니다.

### 4. LL(1) 문법의 엄격한 정의

문법 $G$가 LL(1)이기 위해서는 생성 규칙 $A \rightarrow \alpha | \beta$에 대해 다음 조건들을 만족해야 합니다.

1.  $FIRST(\alpha) \cap FIRST(\beta) = \emptyset$ (두 집합이 서로소여야 함)
2.  만약 $\alpha \Rightarrow^* \epsilon$ 이라면, $FIRST(\beta) \cap FOLLOW(A) = \emptyset$
3.  만약 $\beta \Rightarrow^* \epsilon$ 이라면, $FIRST(\alpha) \cap FOLLOW(A) = \emptyset$

### 5. 주요 키워드 요약

- **Syntax Analysis (구문 분석):** 소스 코드의 구조를 파악하는 단계.
- **Context-Free Grammar (CFG, 문맥 자유 문법):** 프로그래밍 언어의 구문을 정의하는 형식.
- **Recursive Descent Parser (재귀 하강 파서):** 각 비단말 기호를 함수로 구현하여 하향식으로 파싱하는 방식.
- **Look-ahead (미리 보기):** 결정을 내리기 위해 입력 스트림에서 읽어오는 다음 토큰.
- **Undecidable (결정 불가능):** 임의의 CFG를 LL(1)로 변환 가능한지 여부는 이론적으로 판별할 수 없음.

## LL(1) 조건 상세 풀이

세 조건 모두 하나의 공통된 목표를 위한 것입니다. 파서가 현재 토큰 하나만 보고 **"어느 규칙을 써야 하는가"를 혼동 없이 결정**할 수 있어야 한다는 것입니다.

### 조건 1: FIRST(α) ∩ FIRST(β) = {}

**의미:** α와 β 중 어느 것을 선택할지 첫 토큰만 보고 알 수 있어야 한다.

예를 들어 다음 문법을 생각해봅시다.

```
S → a B | a C
```

α = `a B`, β = `a C`이면, 둘 다 `a`로 시작하므로 FIRST(α) ∩ FIRST(β) = {a} ≠ {} 입니다. 파서는 `a`를 보고 `a B`를 써야 할지 `a C`를 써야 할지 알 수 없습니다. 이것이 바로 좌인수분해가 필요한 이유입니다.

반면 다음 문법은 조건 1을 만족합니다.

```
S → a B | b C
```

`a`를 보면 첫 번째 규칙, `b`를 보면 두 번째 규칙을 선택하면 되므로 혼동이 없습니다.

### 조건 2: α →\* ε 이면, FIRST(β) ∩ FOLLOW(A) = {}

**의미:** α가 빈 문자열로 사라질 수 있는 경우, β의 첫 토큰과 A 뒤에 올 수 있는 토큰이 겹치면 안 된다.

이 조건이 필요한 이유를 단계적으로 보겠습니다.

α가 ε으로 유도될 수 있다면, 파서가 현재 토큰 t를 보았을 때 두 가지 해석이 가능해집니다.

- t ∈ FIRST(β)이면 → β 규칙을 써야 한다
- t ∈ FOLLOW(A)이면 → α를 ε으로 소비하고 A 파싱을 종료해야 한다

만약 t가 두 집합에 동시에 속하면 파서는 어느 쪽인지 알 수 없습니다.

구체적인 예시입니다.

```
A → ε | b
```

α = ε, β = `b`이면 FOLLOW(A) = {"b", ...}인 상황을 상상해봅시다. 파서가 `b`를 보았을 때 `A → b`를 적용해야 하는지, 아니면 `A → ε`으로 처리하고 `b`를 A 다음 문맥에서 소비해야 하는지 알 수 없게 됩니다.

### 조건 3: β →\* ε 이면, FIRST(α) ∩ FOLLOW(A) = {}

**의미:** 조건 2와 역할이 대칭적으로, 이번에는 β가 ε으로 사라질 수 있는 경우를 다룹니다.

β가 ε으로 유도될 수 있을 때, 파서가 현재 토큰 t를 보면 두 가지 해석이 가능합니다.

- t ∈ FIRST(α)이면 → α 규칙을 써야 한다
- t ∈ FOLLOW(A)이면 → β를 ε으로 소비하고 A 파싱을 종료해야 한다

마찬가지로 두 집합이 겹치면 결정 불가능한 상황이 됩니다.

### 세 조건의 관계를 한눈에 보면

```
A → α | β 라는 규칙이 있을 때,

1번: α와 β 중 하나를 고를 때 첫 토큰이 겹치지 않아야 함
         (둘 다 실제 토큰으로 시작하는 경우)

2번: α가 사라질 수 있을 때, "β를 시작할 토큰"과
         "A 파싱을 끝낼 토큰"이 겹치지 않아야 함

3번: β가 사라질 수 있을 때, "α를 시작할 토큰"과
         "A 파싱을 끝낼 토큰"이 겹치지 않아야 함
```

결국 세 조건 모두 **"어떤 상황에서도 현재 토큰 하나만 보고 갈림길에서 방향을 결정할 수 있어야 한다"**는 하나의 원칙에서 나온 것입니다.
