## 컴파일러에서의 프론트엔드란?

컴퓨터 과학, 특히 컴파일러 분야에서 **프론트엔드(Frontend)**라는 용어를 사용하는 이유는 웹 개발에서의 의미와는 조금 다릅니다. 컴파일러는 전체 번역 과정을 '단계'별로 나누는데, 그중 **사용자가 작성한 소스 코드와 직접 맞닿아 있는 앞부분**을 프론트엔드라고 부릅니다.

이 용어의 구체적인 의미는 다음과 같습니다:

### 1. 소스 코드의 분석 (사용자 측면)

프론트엔드는 프로그래밍 언어(C, Java, Python 등)로 작성된 소스 코드를 읽어 들여 그 의미를 **분석(Analysis)**하는 역할을 합니다.

- **어휘 및 구문 분석**: 코드의 철자가 맞는지, 문법 구조가 올바른지 확인합니다.

- **의미 분석**: 변수 타입이 맞는지 등 코드의 '의미'가 타당한지 검사합니다.
  이처럼 소스 언어의 특성을 처리하는 단계이기 때문에 '앞단(Frontend)'이라고 부르는 것입니다.

### 2. 중간 표현(IR)으로의 변환

프론트엔드의 가장 큰 임무는 분석한 코드를 **중간 표현(Intermediate Representation, IR)**이라는 공통된 형태로 바꾸는 것입니다.

- 프론트엔드가 코드를 IR로 잘 번역해두면, 뒷부분인 **백엔드(Backend)**는 이 IR을 가져다가 실제 기계어로 바꿉니다.

### 3. 구조적 이점 (N:M 구조)

이렇게 단계를 나누면 매우 효율적입니다:

- **여러 언어 지원**: C용 프론트엔드, Java용 프론트엔드를 각각 만들어도, 결과물을 동일한 **IR**로 뽑아내면 뒷부분(백엔드)은 하나만 있어도 됩니다.

- **여러 기기 지원**: 반대로 백엔드만 교체하면 동일한 소스 코드를 ARM용으로도, x86용으로도 만들 수 있습니다.

즉, 컴파일러에서 **프론트엔드**는 "소스 코드를 해석해서 공통 규격(IR)으로 전달하는 입구" 역할을 하기 때문에 붙여진 이름입니다.

웹 개발에서의 프론트엔드(사용자가 보는 화면)와는 개념이 다르지만, **"사용자(혹은 소스 코드)와 가장 먼저 만나는 부분"**이라는 공통적인 맥락에서 이해하시면 쉽습니다.

## 컴파일러의 동작 단계

컴파일러는 우리가 작성한 소스 코드를 컴퓨터가 이해할 수 있는 기계어로 번역하기 위해 크게 **분석(Analysis)**과 **합성(Synthesis)** 과정을 거칩니다. 각 단계의 흐름은 다음과 같습니다.

## 1. 프론트엔드 (분석 단계)

소스 코드의 구조와 의미를 파악하여 공통된 형태인 중간 표현(IR)으로 만듭니다.

- **어휘 분석 (Lexical Analysis)**: 소스 코드를 '토큰(Token)'이라는 의미 있는 최소 단위로 쪼갭니다.

- _예: `position` → 식별자, `+` → 연산자_

- **구문 분석 (Syntactic Analysis)**: 토큰들이 언어의 문법에 맞는지 확인하고, 이를 트리 형태인 **추상 구문 트리(AST)**로 구조화합니다.

- **의미 분석 (Semantic Analysis)**: 타입 체크(Type Checking) 등을 통해 프로그램의 의미가 타당한지 검사합니다.

- 예: 정수와 실수를 더할 때 자동으로 형 변환(int2float)을 수행하도록 정보를 추가함

## 2. 미들엔드 (최적화 단계)

특정 언어나 하드웨어에 종속되지 않은 상태에서 코드의 효율성을 높입니다.

- **중간 코드 생성 (Intermediate Code Generation)**: 분석된 결과를 바탕으로 기계 독립적인 **중간 표현(IR)**을 생성합니다.

- **코드 최적화 (Code Optimizer)**: 불필요한 계산을 줄여 실행 속도를 높이거나 메모리 사용량을 줄입니다.

- **상수 전파**: `3 * 1` 같은 식을 미리 계산하여 상수로 대체함.

- **루프 불변 코드 이동**: 반복문 안에서 변하지 않는 식을 밖으로 빼내어 중복 계산을 방지함.

## 3. 백엔드 (합성 단계)

최적화된 코드를 실제 하드웨어가 실행할 수 있는 형태로 바꿉니다.

- **코드 생성 (Code Generation)**: 중간 코드를 대상 아키텍처(x86, ARM 등)에 맞는 **어셈블리 코드**나 **기계어**로 변환합니다.

- **레지스터 할당**: CPU의 한정된 자원인 레지스터를 어떤 변수에 할당할지 결정합니다.

### 요약: 컴파일러의 컨베이어 벨트

1.  **입력**: 고수준 언어 소스 코드

2.  **프론트엔드**: "이 코드가 문법에 맞는가? 무슨 뜻인가?" (분석)

3.  **미들엔드**: "더 효율적으로 바꿀 수 있는가?" (최적화)

4.  **백엔드**: "이 기계(CPU)가 이해하게 만들자!" (생성)

5.  **출력**: 타겟 어셈블리/기계어 프로그램

이 과정 중 하나라도 오류(Syntax Error 등)가 발견되면 컴파일러는 번역을 중단하고 사용자에게 알립니다.

- 프로그램의 **번역(Translation)**과 **해석(Interpretation)**의 차이를 다룹니다.

### [2단계] 어휘 분석 (Lexical Analysis)

- 소스 코드를 **토큰(Token)** 단위로 쪼개는 방법을 배웁니다.

- 이 과정에서 필요한 **정규 표현식(Regular Expressions)**과 **오토마타(Automata)** 이론을 다룹니다.

### [3단계] 구문 분석 (Syntactic Analysis)

- 프로그램의 구조를 분석합니다.

- **문맥 자유 문법(CFG)**, 재귀 하강 파서(Recursive Descent Parser), **추상 구문 트리(AST)**의 생성 원리를 학습합니다.

### [4단계] 의미 분석 (Semantic Analysis)

- 프로그램이 문법적으로 맞는지뿐만 아니라, **의미(Meaning)**상 오류가 없는지 확인합니다.

- 속성 문법(Attribute Grammars)과 **타입 체크(Type Checking)**가 포함됩니다.

### [5단계] 런타임 조직 (Run-time Organization)

- 프로그램이 실행될 때 데이터가 메모리에 어떻게 배치되는지 배웁니다.

- **정적 할당, 스택(Stack), 힙(Heap)** 메모리 구조를 다룹니다.

### [6단계] 코드 생성 및 최적화 (Code Generation)

- 특정 아키텍처를 위한 최종 코드를 생성하는 법을 배웁니다.

- **Java 가상 머신(JVM)과 Java 바이트코드**를 이해하고, 코드 최적화 기법을 학습합니다.

## 3. 요약 및 시사점

이 과목은 소스 코드가 입력되어 기계어로 번역되기까지의 전 과정을 **이론적으로 분석하고 실제로 구현**해보는 과정입니다.

- **평가 방식**: 앞서 언급되었듯, 이러한 원리들을 적용하여 짧은 테스트 케이스를 통과시키는 프로젝트 위주로 진행될 것입니다.

- **기대 효과**: 컴파일러 기법뿐만 아니라 프로그래밍 언어의 구문, 의미, 변수 바인딩 등에 대한 깊은 이해를 얻을 수 있습니다.

## 정적 의미론(Static Semantics)과 동적 의미론(Dynamic Semantics)의 차이점

두 개념의 가장 큰 차이는 **"오류를 언제 검사하느냐"**에 있습니다.

### 1. 정적 의미론 (Static Semantics)

컴파일러가 프로그램을 실제 실행 파일로 만들기 전인 **컴파일 시간(Compile-time)**에 검사하는 규칙들입니다.

- **검사 시점**: 코드 번역 단계(의미 분석 단계)에서 수행됩니다.

- **주요 검사 내용**:
- **타입 체크(Type checking)**: 변수에 잘못된 자료형의 값을 넣지는 않았는지 검사합니다.

- **서브루틴 인자 검사**: 함수를 호출할 때 인자의 개수나 형식이 맞는지 확인합니다.

- **식별자 사용**: 선언되지 않은 변수를 쓰거나, 현재 범위(Scope)에서 사용할 수 없는 이름을 쓰는지 체크합니다.

### 2. 동적 의미론 (Dynamic Semantics)

컴파일 시간에는 확인하기 너무 복잡하거나 불가능하여, 프로그램이 **실행 중(Run-time)**에 검사하게 되는 규칙들입니다.

- **검사 방법**: 컴파일러가 실행 시점에 이 규칙들을 체크할 수 있는 **특수한 코드(Checks)**를 프로그램 내부에 삽입합니다.

- **주요 검사 내용**:
- **배열 인덱스 범위(Array subscript bounds)**: 배열의 크기를 벗어난 곳에 접근하려 하는지 실행 중에 감시합니다.

- **산술 오류**: 0으로 나누기(Division by zero), 오버플로우/언더플로우 등을 잡아냅니다.

- **초기화되지 않은 변수**: 값이 할당되지 않은 변수를 사용하려 할 때 발생하는 오류를 체크합니다.

### 요약 비교

| 구분          | 정적 의미론 (Static)    | 동적 의미론 (Dynamic) |
| ------------- | ----------------------- | --------------------- |
| **체크 시점** | 컴파일 시간 (번역할 때) |

| 실행 시간 (실행될 때)

|
| **수행 주체** | 컴파일러 자체

| 컴파일러가 삽입한 체크 코드

|
| **특징** | 실행 전에 미리 오류를 발견 가능

| 실행을 해봐야만 알 수 있는 복잡한 오류

|

강의 자료에서 언급된 **'의미 분석(Semantic Analysis)'** 단계는 주로 이 중 **정적 의미론**을 다루는 단계라고 이해하시면 됩니다.

## Syntax(구문론)와 Semantics(의미론)의 차이점

### 1. Syntax (구문론): 프로그램의 형태와 구조

Syntax는 프로그램이 어떻게 보이는지, 즉 **문장이나 식의 형식적인 구조**를 다룹니다.

- **관심 분야**: 프로그램의 '의미'에는 관심이 없으며, 오직 **구조가 올바른지**만 따집니다.

- **표현 방법**: 주로 **문맥 자유 문법(Context-Free Grammar, CFG)**을 사용하여 명시합니다.

- **예시**: `if` 뒤에 괄호가 있는지, 문장 끝에 세미콜론(`;`)이 있는지 등 '글자들의 배치 규칙'이 여기에 해당합니다.

### 2. Semantics (의미론): 프로그램의 의미

Semantics는 해당 구문이 나타내는 **실제 동작이나 의미**를 다룹니다. 프로그램이 실행될 때 어떤 일이 일어나는지를 결정합니다.

- **표현 방법**: 속성 문법(Attribute Grammars)이나 자연어 설명 등을 통해 정의됩니다.

- **세부 분류** (43페이지 참고):

- **정적 의미론 (Static Semantics)**: 컴파일 시점에 체크되는 규칙입니다. 예를 들어, 변수의 타입이 맞는지(Type checking) 확인하는 작업이 포함됩니다.

- **동적 의미론 (Dynamic Semantics)**: 실행 시점에 체크되는 규칙입니다. 0으로 나누기 오류나 배열 범위를 벗어난 접근 등을 다룹니다.

3. 요약 비교

| 구분          | Syntax (구문)           | Semantics (의미) |
| ------------- | ----------------------- | ---------------- |
| **핵심 질문** | "문법에 맞는 형태인가?" |

| "어떤 동작을 수행하는가?"

|
| **비유** | 한국어의 '맞춤법'과 '문장 구조'

| 문장이 전달하고자 하는 '뜻'

|
| **정의 도구** | CFG (Context-Free Grammar)

| Attribute Grammars 등

|
| **오류 예시** | <br>`;` 누락, 괄호 불일치

| 정수 변수에 문자열 대입, 0으로 나누기

|

컴파일러는 먼저 **Syntax**를 분석하여 나무 모양의 구조(AST)를 만들고, 그 후에 **Semantics**를 분석하여 해당 코드의 타입이 맞는지, 실제 실행 가능한 로직인지를 검사하는 과정을 거칩니다.

## verbose

`gcc -v helloworld.c` 명령어는 컴파일러가 소스 코드를 실행 파일로 만드는 **전체 과정을 상세하게 출력(Verbose)**하라는 명령입니다.

이 명령을 실행하면 단순히 파일을 생성하는 것을 넘어, 내부적으로 어떤 도구들이 어떤 순서로 작동하는지 확인할 수 있습니다. 주요 역할은 다음과 같습니다.

### 1. 컴파일 단계별 흐름 확인

컴파일러는 내부적으로 여러 도구를 거쳐 실행 파일을 만드는데, `-v` 옵션을 통해 이 흐름을 눈으로 확인할 수 있습니다:

- **전처리기(Preprocessor)**: `#include`나 `#define` 등을 처리하여 확장된 소스 코드를 만듭니다.

- **컴파일러(Compiler)**: 소스 코드를 대상 어셈블리 프로그램으로 번역합니다.

- **어셈블러(Assembler)**: 어셈블리 코드를 기계가 이해할 수 있는 재배치 가능한 기계어(Object 파일)로 바꿉니다.

- **링커(Linker)**: 여러 오브젝트 파일과 라이브러리를 하나로 묶어 최종 실행 파일을 만듭니다.

### 2. 상세 정보 제공

- **버전 확인**: 사용 중인 `gcc`의 정확한 버전 정보를 보여줍니다.

- **경로 및 환경 설정**: 컴파일에 필요한 헤더 파일(`.h`)이나 라이브러리(`.a`, `.so`)를 찾기 위해 컴파일러가 어떤 디렉토리를 탐색하는지 보여줍니다.
- **내부 명령어 노출**: 실제로 호출되는 내부 프로그램(예: `cc1`, `as`, `ld`)과 그 프로그램들에 전달되는 복잡한 옵션들을 모두 출력합니다.

### 왜 사용하나요?

보통은 컴파일이 잘 안 될 때 **디버깅 용도**로 사용합니다. "내가 설치한 라이브러리를 왜 못 찾지?" 혹은 "어떤 헤더 파일을 참조하고 있는 거지?"라는 의문이 생겼을 때 이 명령어를 써보면 컴파일러가 내부적으로 수행하는 모든 작업을 낱낱이 보여주기 때문에 문제 원인을 찾기 수월해집니다.

## 소 그림에서 반도체는 ISA를 뜻하는 거구나

네, 정확합니다! 소 그림(슬라이드 49페이지)의 **BACK(백엔드)** 부분에 있는 반도체 모양의 아이콘은 특정 하드웨어의 설계 규격인 **ISA(Instruction Set Architecture, 명령어 세트 아키텍처)**를 의미합니다.

이 그림이 전달하는 핵심 메시지는 다음과 같습니다.

### 1. ISA는 소프트웨어와 하드웨어의 약속

- 그림에 표시된 **ARM**이나 **x86**이 바로 대표적인 ISA의 종류입니다.

- 컴파일러의 백엔드는 중간 표현(IR)을 가져와서, 특정 반도체(CPU)가 이해할 수 있는 고유의 명령어 집합인 ISA에 맞춰 어셈블리 코드를 생성합니다.

### 2. 기기별 최적화의 핵심

- 똑같은 소스 코드로 시작했더라도, 결과물이 **Intel/AMD CPU**에서 돌아가려면 **x86 ISA**로, **아이폰이나 갤럭의 CPU**에서 돌아가려면 **ARM ISA**로 번역되어야 합니다.

- 소 그림에서 백엔드 부위가 여러 아키텍처(ARM, x86)로 나뉘어 있는 이유도 바로 이 때문입니다.

### 3. 컴파일러의 'Synthesis' 과정

- 슬라이드 50페이지를 보면 이 백엔드 단계를 **Synthesis(합성)**라고 부르기도 합니다.

- 분석(Analysis)된 논리(IR)를 바탕으로, 실제 반도체 칩이라는 물리적 환경에 맞는 실행 코드를 '조립'해내는 과정이기 때문입니다.

결국 **"IR이라는 공통 레시피를 각 반도체(ISA) 전용 언어로 통역하는 곳"**이 바로 소의 뒷다리(백엔드) 부분이라고 이해하시면 완벽합니다.

## 어휘 분석(Lexical Analysis)에서 ast를 만들어?

아니요, **어휘 분석(Lexical Analysis) 단계에서는 AST를 만들지 않습니다.**

강의 자료 52페이지와 8페이지의 흐름도를 보면, 컴파일러는 각 단계마다 생성하는 결과물이 명확히 정해져 있습니다.

### 1. 어휘 분석(Lexical Analysis)의 결과물: 토큰(Tokens)

- 이 단계에서는 소스 코드를 읽어 의미 있는 최소 단위인 **토큰**으로 쪼개기만 합니다.

- 예를 들어 `position = initial + rate * 60`을 `<id, 1>`, `<=>`, `<id, 2>`, `<+>` 같은 **선형적인 줄(Stream)**로 만드는 것이 목적입니다.

### 2. AST(추상 구문 트리)는 언제 만드나요?

- **구문 분석(Syntactic Analysis/Parsing)** 단계에서 만듭니다.

- 어휘 분석기가 넘겨준 **토큰들**을 재료로 삼아, 문법 규칙(CFG)에 따라 계층적인 나무 구조인 **AST**를 조립해내는 것입니다.

요약하자면

| 단계                   | 입력              | 결과물 |
| ---------------------- | ----------------- | ------ |
| **어휘 분석(Lexical)** | 소스 코드(문자열) |

| <br>**토큰(Tokens)**

|
| **구문 분석(Syntax/Parsing)** | 토큰(Tokens)

| <br>**AST(추상 구문 트리)**

|
| **의미 분석(Semantic)** | AST

| <br>**장식된(Decorated) AST**

|

즉, 어휘 분석기는 나무(AST)를 만드는 것이 아니라 나무를 만들 때 쓸 **벽돌(토큰)**을 준비하는 역할이라고 이해하시면 됩니다.

## 아 x86, risc-v처럼 독자적인 isa를 갖는구나

네, 정확합니다. **JVM은 물리적인 CPU와 마찬가지로 자신만의 독자적인 명령어 세트(Instruction Set)를 가진 가상의 프로세서**라고 이해하시면 완벽합니다.

- **추상화된 실행**: 이 바이트코드는 실제 하드웨어가 아닌 **JVM이라는 가상 머신 위에서 실행**되도록 설계된 전용 언어입니다.

### 2. 하드웨어 아키텍처와의 비교

JVM은 실제 CPU의 동작 방식을 소프트웨어로 흉내 냅니다.

| 구분            | 실제 CPU (x86, RISC-V 등) | 가상 머신 (JVM) |
| --------------- | ------------------------- | --------------- |
| **명령어 세트** | 기계어 (Native Code)      |

| <br>**Java 바이트코드**

|
| **데이터 처리** | 레지스터 기반 (Registers)

| <br>**스택 기반 (Stack-based)**

|
| **메모리 관리** | 물리 메모리 직접 관리

| 가상 메모리 영역 (Heap, Stack 등)

|

### 3. 컴파일러 프로젝트에서의 의미 (Target Machine)

- **최종 타겟**: 이번 학기 프로젝트에서 여러분의 **Code Generator**가 뱉어내야 할 최종 결과물은 x86용 `.exe` 파일이 아니라, JVM ISA를 따르는 **`.class` 파일(바이트코드)**입니다.

- **범용성**: 여러분이 만든 컴파일러로 바이트코드를 한 번만 생성해두면, JVM이 설치된 x86 PC든 ARM 맥북이든 어디서나 동일하게 실행됩니다.

결국 **JVM은 "소프트웨어로 만든 CPU"**이며, 여러분은 이번 학기에 그 가상의 CPU를 위한 **전용 언어(바이트코드) 통역사**를 만들게 되는 것입니다.

## 이건 이해를 돕기 위해서 IR이 아니라 C로 해놓은거라고 하시네

네, 교수님 말씀이 맞습니다. 강의 자료 56페이지의 **코드 최적화(Code Optimization)** 예시는 학생들이 최적화의 원리를 더 직관적으로 이해할 수 있도록 **중간 표현(IR) 대신 우리에게 익숙한 C 언어 형태**를 빌려 설명하고 있는 것입니다.

실제 컴파일러 내부에서는 이 과정이 다음과 같은 논리로 진행됩니다.

### 1. 왜 IR 대신 C로 보여주나요?

- **가독성**: 실제 IR(예: LLVM IR)은 `alloca`, `load`, `store` 같은 저수준 명령어로 가득 차 있어 한눈에 흐름을 파악하기 어렵습니다.

- **최적화 개념 전달**: 상수 전파(Constant propagation)나 불필요한 코드 제거(Dead code elimination) 같은 개념은 소스 코드 수준에서 보여줄 때 훨씬 명확하게 전달됩니다.

### 2. 슬라이드 56페이지의 최적화 단계 요약

이 예시는 총 3번의 과정(Pass)을 거치며 코드가 어떻게 다듬어지는지 보여줍니다.

- **Pass 1 (상수 전파)**: 변수 `a`가 1이라는 것을 알기 때문에, `a+9`를 `10`으로, `3*a`를 `3`으로 미리 계산해 버립니다.

- **Pass 2 (데드 코드 제거)**: `if (10 > 100)`은 항상 거짓(False)이므로, `then` 절에 있는 코드는 절대 실행될 일이 없습니다. 따라서 이 부분을 통째로 삭제합니다.

- **Pass 3 (2차 상수 전파)**: 앞선 단계에서 코드가 정리된 덕분에, 마지막에 있는 `foo(a+1)`도 `foo(2)`로 바꿀 수 있게 됩니다.

### 3. 실제 컴파일러에서는?

실제로는 **슬라이드 49페이지의 'MIDDLE' 부위**에서 이 작업이 일어납니다.

1. 프론트엔드가 만든 **IR 코드**를 입력으로 받습니다.

2. **LLVM Optimizer**와 같은 도구가 수십 개의 Pass(Pass 1, 2, ... N)를 돌리며 위와 같은 논리로 코드를 줄입니다.

3. 그 결과물로 훨씬 가볍고 빠른 **최적화된 IR**이 나옵니다.

결국 이 슬라이드는 **"컴파일러가 우리가 짠 비효율적인 코드를 어떻게 논리적으로 분석해서 똑똑하게 고치는지"** 그 핵심 원리를 보여주는 아주 좋은 예시입니다.
