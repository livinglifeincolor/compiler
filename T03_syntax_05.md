## 주요 내용 정리

### 1. 구문 트리(Parse Tree) vs. 추상 구문 트리(AST)

- **구문 트리 (Concrete Syntax Tree):** 입력 소스 코드의 구체적인 문법 구조를 그대로 반영합니다. 유도(Derivation) 과정과 1대1로 대응하며, 파싱을 위한 보조 비단말 기호(T, F 등)나 구두점(괄호, 세미콜론)을 모두 포함하여 구조가 복잡합니다.
- **추상 구문 트리 (AST):** 컴파일러의 후속 단계(타입 체크, 최적화, 코드 생성)에 필요한 핵심 정보만을 추출한 트리입니다. 문법적 상세 내용(구두점, 키워드, 연산자 우선순위 처리를 위한 중간 노드)을 생략하여 더 압축된 형태를 가집니다.

### 2. AST 설계 및 구현 (MiniC 중심)

- **클래스 설계:** Python 클래스를 사용하여 AST의 각 요소를 정의합니다. 상위 추상 클래스(`AST`)를 두고, 이를 상속받아 구체적인 클래스(`AssignStmt`, `BinaryExpr`, `IntLiteral` 등)를 구성합니다.
- **단말 노드 (Terminal Nodes):** AST에서는 소스 코드의 구두점이나 키워드를 별도 노드로 저장할 필요가 없습니다. 주로 식별자(ID), 리터럴, 연산자 등을 단말 노드로 처리합니다.
- **AST 구축 과정:** 기존의 재귀 하강 파서(Recursive Descent Parser)의 각 파싱 함수에 AST 클래스의 생성자 호출 코드를 삽입하여 파싱과 동시에 트리를 생성합니다.

### 3. 주요 도구 및 패턴

- **Visitor Design Pattern:** 생성된 AST를 순회하며 화면에 그리거나(`TreeDrawer`), 텍스트로 출력하거나(`TreePrinter`), 다시 코드로 변환(`UnParser`)하는 작업에 사용됩니다.
- **컴파일러 옵션:** MiniC 컴파일러는 `-ast`(트리 표시), `-astp`(소스 위치 포함), `-u`(언파싱) 등의 옵션을 통해 AST 정보를 확인하는 기능을 제공합니다.

### 4. 촘스키 계층 (Chomsky's Hierarchy)

언어를 기술하는 문법의 복잡도에 따라 4가지 유형으로 분류합니다.

| 유형       | 문법 이름              | 특징 및 용도                                 |
| :--------- | :--------------------- | :------------------------------------------- |
| **Type 0** | Unrestricted           | 제약이 없는 문법                             |
| **Type 1** | Context-sensitive      | 대부분의 프로그래밍 언어 특성이 여기에 속함  |
| **Type 2** | **Context-free (CFG)** | **파서(Parser)** 구현에 사용 (LL, LR 파싱)   |
| **Type 3** | **Regular**            | **스캐너(Scanner)** 구현 및 토큰 정의에 사용 |

### 5. 문법의 한계

- **정규 문법:** 중첩된 구조(Nested constructs)를 표현할 수 없으며, 개수를 세는 능력이 없습니다.
- **문맥 자유 문법 (CFG):** 변수 선언 후 사용 여부 확인이나 함수 호출 시 인자 개수 일치 확인과 같은 제약 조건은 CFG로 표현할 수 없습니다. 이러한 항목들은 **의미 분석(Semantic Analysis)** 단계에서 체크해야 합니다.

## Chomsky's Hierarchy

### 1. 계층별 상세 분석

#### **Type 0: 무제약 문법 (Unrestricted Grammars)**

가장 일반적인 형태의 문법으로, 생성 규칙에 실질적인 제약이 없습니다.

- **생성 규칙**: $\alpha \rightarrow \beta$ (단, $\alpha \neq \epsilon$)
- **특징**: 좌변과 우변에 터미널(Terminal)과 비단말(Non-terminal) 기호의 임의의 조합이 올 수 있습니다.
- **컴퓨팅 모델**: 튜링 기계(Turing Machine)와 동등한 계산 능력을 갖추며, 해당 언어에 대한 인식 문제는 **반판정 가능(Semi-decidable)**합니다.

#### **Type 1: 문맥 의존 문법 (Context-Sensitive Grammars)**

치환되는 기호 주변의 문맥(Context)을 고려해야 하는 문법입니다.

- **생성 규칙**: $\alpha A \gamma \rightarrow \alpha \beta \gamma$
- **제약**: 비단말 기호 $A$가 $\beta$로 치환되기 위해서는 좌우에 $\alpha$와 $\gamma$라는 문맥이 유지되어야 합니다. 또한, 우변의 길이가 좌변보다 짧아질 수 없습니다($|\beta| \geq 1$).
- **시간 복잡도**: **PSPACE-complete** 범주에 속하며, 매우 높은 계산 비용을 요구합니다.

#### **Type 2: 문맥 자유 문법 (Context-Free Grammars)**

비단말 기호가 주변 문맥과 상관없이 독립적으로 치환되는 문법입니다.

- **생성 규칙**: $A \rightarrow \alpha$
- **제약**: 좌변에는 반드시 **단 하나의 비단말 기호**만 존재해야 합니다.
- **컴파일러 적용**: 프로그래밍 언어의 구문 구조(Syntax)를 정의하는 데 핵심적으로 사용됩니다.
- **성능**: 일반적인 경우 $O(n^3)$의 시간 복잡도를 가지나, 컴파일러에서 사용하는 **LL, LR, LALR** 문법과 같은 하위 부분집합은 **$O(n)$ (선형 시간)** 내에 파싱이 가능합니다.

#### **Type 3: 정규 문법 (Regular Grammars)**

가장 엄격한 제약을 가지며, 유한 오토마타로 인식 가능한 언어입니다.

- **생성 규칙**: $A \rightarrow a$ 또는 $A \rightarrow aB$
- **제약**: 비단말 기호는 하나의 단말 기호를 생성하거나, 단말 기호와 비단말 기호의 쌍으로만 전개될 수 있습니다.
- **컴파일러 적용**: 소스 코드의 최소 단위인 **토큰(Token)**을 추출하는 **어휘 분석(Lexing)** 단계에서 사용됩니다.
- **성능**: $O(n)$의 최악 시간 복잡도를 보장합니다.

### 2. 컴파일러 설계에서의 실질적 의미

컴파일러는 전체 언어를 하나의 문법으로 처리하지 않고, 효율성을 위해 계층을 나누어 처리합니다.

- **어휘 분석기(Scanner)**: 정규 문법(Type 3)을 사용하여 식별자, 예약어, 리터럴 등을 빠르게 인식합니다.
- **구문 분석기(Parser)**: 문맥 자유 문법(Type 2)을 사용하여 프로그램의 구조적 올바름을 확인하고 추상 구문 트리(AST)를 구축합니다.
- **의미 분석(Semantic Analysis)**: 문맥 자유 문법으로 표현할 수 없는 제약 조건(예: 변수 선언 전 사용 여부, 타입 일치 등)은 이 단계에서 별도의 심볼 테이블과 로직을 통해 처리합니다. 이는 실제 프로그래밍 언어가 기술적으로는 **문맥 의존적(Context-sensitive)** 성격을 띠기 때문입니다.

## Limitations of Regular Grammars and Context-Free Grammars

### 1. Regular Grammars (RG)의 한계

정규 문법은 **유한 오토마타(Finite Automata)**로 구현될 수 있을 만큼 단순합니다. 하지만 그 단순함이 곧 독이 되기도 하죠.

- **메모리(상태)의 부재 (Counting의 한계):**
  정규 문법은 "기억"을 할 수 없습니다. 예를 들어, 괄호가 열린 만큼 닫혀야 하는 $a^n b^n$ 형태의 언어를 인식하지 못합니다. $n$이 얼마인지 세어둘 공간이 없기 때문이죠.
- **중첩 구조(Nested Structure) 표현 불가:**
  프로그래밍 언어의 핵심인 중첩된 `if-else`나 재귀적인 수식 구조를 표현할 수 없습니다.
- **Pumping Lemma의 제약:**
  정규 언어라면 반드시 만족해야 하는 'Pumping Lemma'를 만족하지 못하는 언어들이 많습니다. 즉, 패턴이 반복된다고 해서 무한히 확장할 수 없는 복잡한 논리는 정규 문법의 영역 밖입니다.

### 2. Context-Free Grammars (CFG)의 한계

CFG는 괄호 짝 맞추기나 중첩 구조는 기가 막히게 처리합니다. 하지만 '문맥(Context)'을 무시한다는 점이 발목을 잡습니다.

- **다중 카운팅 및 동기화 불가:**
  $a^n b^n$은 가능하지만, $a^n b^n c^n$처럼 세 가지 이상의 요소 개수를 맞추는 것은 불가능합니다. 스택(Pushdown Automata) 하나만으로는 두 개 이상의 대상을 동시에 추적하기 어렵기 때문입니다.
- **식별자 선언 확인 (Context-Sensitivity):**
  가장 치명적인 한계입니다. 프로그램에서 **"변수가 사용되기 전에 선언되었는가?"**를 CFG 수준에서는 확인할 수 없습니다. 변수 이름($w$)이 나중에 다시 등장($w$)하는 형태($ww$ 패턴)는 CFG가 처리할 수 없는 영역입니다.
- **모호성(Ambiguity) 문제:**
  하나의 문장에 대해 여러 개의 파스 트리(Parse Tree)가 만들어지는 경우가 생깁니다. 물론 문법을 수정해 해결하기도 하지만, 어떤 언어는 본질적으로 모호함을 제거할 수 없는 경우(Inherently Ambiguous)도 존재합니다.

### 요약 비교 테이블

| 구분                     | Regular Grammars (RG)          | Context-Free Grammars (CFG)           |
| :----------------------- | :----------------------------- | :------------------------------------ |
| **주요 한계**            | **메모리 없음.** (카운팅 불가) | **문맥 참조 불가.** (의미론적 제약)   |
| **대표적인 불가능 예시** | $a^n b^n$, 중첩된 괄호         | $a^n b^n c^n$, 변수 선언 전 사용 확인 |
| **대응 기계**            | 유한 오토마타 (FA)             | 푸시다운 오토마타 (PDA)               |
| **컴파일러에서의 역할**  | 어휘 분석 (Lexical Analysis)   | 구문 분석 (Syntax Analysis)           |

결국 이런 한계들 때문에 우리는 컴파일 과정에서 CFG로 구문 분석을 마친 뒤, **심볼 테이블(Symbol Table)**을 활용하거나 **Context-Sensitive**한 분석 단계를 추가로 거쳐 프로그램의 무결성을 검증하게 됩니다.
