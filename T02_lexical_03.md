## 키워드 정리

### 1. 스캐너 생성 도구 (Scanner Generators)

어휘 분석기를 수동으로 작성하는 대신, 특정 사양(Specification)을 입력하면 자동으로 코드를 생성해 주는 도구들입니다.

- **C 기반:** `lex` (UNIX), `flex` (GNU의 빠른 버전).
- **Java 기반:** `JLex`, `JFlex`, `JavaCC`.
- **JLex 구조:** 사용자 코드, JLex 지시문(Declarations), 정규 표현식 규칙(Regular Expression Rules)의 세 부분으로 구성되며, 각 섹션은 `%%`로 구분합니다.

### 2. 구현 방식: Table-driven vs. Direct-coded

스캐너가 입력 문자열을 인식하는 두 가지 주요 아키텍처입니다.

- **Table-driven (테이블 구동 방식):** 자동화된 생성 도구(JLex 등)에서 주로 사용합니다. 오토마타를 거대한 상태 전이 테이블로 인코딩하며, 구현이 쉽고 자동화에 적합하지만 테이블 크기가 커지면 캐시 효율이 떨어질 수 있습니다.
- **Direct-coded (직접 코드 방식):** 오토마타의 상태 전이를 프로그램 카운터(제어 흐름)에 직접 인코딩합니다. `switch-case`나 `if-else` 문을 사용하며, 테이블 룩업이 필요 없어 일반적인 CPU에서 더 빠른 성능을 보입니다.

### 3. 정규 표현식과 오토마타의 변환 과정

스캐너 생성기가 정규 표현식(RE)으로부터 효율적인 인식기를 만드는 단계적 알고리즘입니다.

1.  **RE → NFA:** Thompson의 구성을 통해 정규 표현식을 비결정적 유한 오토마타(NFA)로 변환합니다.
2.  **NFA → DFA:** **Subset Construction(부분 집합 구성)** 알고리즘을 사용하여 NFA를 결정적 유한 오토마타(DFA)로 변환합니다. 이 과정에서 `ε-closure`와 `Move` 함수가 핵심적인 역할을 합니다.
3.  **DFA → Minimal-state DFA:** 불필요한 상태를 제거하여 최소 상태를 가진 DFA로 최적화합니다.

### 4. 정규 언어의 한계 (Limitations)

정규 표현식은 프로그래밍 언어의 모든 요소를 표현할 수 없습니다.

- **표현 가능:** 키워드, 식별자, 리터럴, 연산자, 구두점 등.
- **표현 불가능:** $0^{n}1^{n}$과 같이 개수가 일치해야 하는 패턴이나 산술식의 괄호 쌍 맞추기 등 **Self-embedding(자기 내포)** 구조는 정규 표현식으로 처리할 수 없습니다.
- **해결책:** 이러한 복잡한 구문(Expressions, Statements 등)을 정의하기 위해서는 정규 표현식보다 강력한 **문맥 자유 문법(Context-Free Grammars, CFGs)**이 필요합니다.

### 5. 어휘 분석의 규칙

패턴 매칭 시 발생할 수 있는 모호성을 해결하는 두 가지 원칙입니다.

- **우선순위 (Priority):** 여러 패턴이 동시에 일치할 경우, 사양 파일에서 먼저 정의된 규칙을 우선합니다. (예: `if`를 식별자가 아닌 키워드로 인식)
- **최장 일치 (Longest Match):** 입력 문자열에서 매칭될 수 있는 가장 긴 토큰을 선택합니다. (예: `<=`를 `<`와 `=` 두 개가 아닌 하나의 토큰으로 인식)

## on the fly (pdf page 22)

### 1. On-the-fly simulation의 의미

이 방식은 정규 표현식을 미리 DFA(결정적 유한 오토마타)로 변환해두지 않고, 입력을 읽으면서 **그때그때 실시간으로** NFA 상태를 추적하는 방식입니다.

- **동작 방식**: 문자열을 한 글자씩 읽을 때마다 현재 위치에서 도달 가능한 NFA의 모든 상태 집합을 병렬적으로 계산합니다.
- **유용한 사례**: 텍스트 에디터나 IDE에서 일회성으로 수행하는 검색(Find) 기능처럼, 정규 표현식이 한 번만 사용되는 시나리오에 적합합니다.

### 2. 컴파일러에서 이 방식이 비효율적인 이유

이미지의 두 번째 문장은 컴파일러의 특수성을 설명하고 있습니다.

- **반복 사용**: 컴파일러는 프로그래밍 언어의 토큰 규칙(키워드, 식별자 등)을 정의한 정규 표현식을 소스 코드를 읽는 동안 **수없이 반복해서** 사용해야 합니다.
- **속도 저하**: On-the-fly 방식은 매번 다음 상태를 계산해야 하는 오버헤드가 발생하므로, 대량의 코드를 처리해야 하는 컴파일러에게는 속도가 너무 느립니다.

### 3. 더 효율적인 접근법 (More efficient approach)

이미지 끝부분의 화살표가 가리키는 '더 효율적인 접근법'은 바로 **미리 DFA를 생성하는 것**입니다.

- **DFA 변환**: 정규 표현식을 NFA로 만든 뒤, 이를 다시 DFA로 변환하여 고정된 전이 테이블(Transition Table)을 생성합니다.
- **상태 고정**: 이미 모든 전이 경로가 계산되어 있으므로, 스캐너는 테이블 룩업만으로 매우 빠르게 토큰을 인식할 수 있습니다.

**요약하자면**, On-the-fly는 "그때그때 계산하느라 준비 시간은 짧지만 실행은 느린 방식"이고, 컴파일러가 선호하는 방식은 "미리 계산(DFA)해두느라 준비 시간은 걸리지만 실행은 압도적으로 빠른 방식"입니다.
